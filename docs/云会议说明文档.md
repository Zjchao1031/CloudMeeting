## 1. 引言

### 1.1 产品概述

**云会议桌面客户端**（CloudMeeting Client）是一款运行于 Windows 11 平台的桌面端实时通信应用程序。用户无需注册账号，即可快速创建或加入在线会议，进行多人音视频通话、屏幕共享与文字聊天。客户端通过网络与部署在 Ubuntu 24 服务器上的后端服务进行通信，实现媒体流的传输与信令交互。

本产品定位于**轻量、快速、免登录**的会议场景，适合团队内部临时协作、远程演示、小型在线研讨等使用场景。

### 1.2 目标用户

- 需要快速发起临时会议的团队负责人或个人用户；
- 参与远程协作、需要共享屏幕或实时沟通的工作人员；
- 对隐私要求较高、不希望强制注册账号的用户。

## 3. 功能模块详解

### 3.1 用户信息设置

#### 功能描述

用户在进入主界面后，可通过**设置面板**配置个人信息。本模块无需网络连接，所有数据均保存在本地。

#### 支持的设置项


| 设置项 | 说明                                                               |
| --- | ---------------------------------------------------------------- |
| 昵称  | 单行文本输入框，长度限制 32 个字符，不可为空                                         |
| 头像  | 支持从本地选择图片文件（JPG/PNG），客户端自动裁剪为正方形并缩放为 64×64 像素；未设置时显示内置默认头像 |


#### 数据持久化

- 昵称与头像以本地文件形式保存（如应用数据目录下的配置文件），应用重启后自动加载；
- 用户也可在创建或加入会议时临时设定昵称与头像，临时设定不覆盖本地保存的配置。

---

### 3.2 会议管理

#### 3.2.1 创建会议

用户在主界面点击**「创建会议」**按钮后，弹出会议初始化配置面板。

**配置项说明：**


| 配置项      | 控件类型                    | 默认值 | 说明                    |
| -------- | ----------------------- | --- | --------------------- |
| 最大参会人数   | 数值选择框（SpinBox）          | 10  | 可选范围：2 ~ 100 人        |
| 是否启用密码   | 开关（Toggle Switch）       | 关闭  | 开启后显示密码输入框            |
| 会议密码     | 密码输入框（PasswordLineEdit） | 空   | 仅在启用密码开关后可见；长度 4~16 位 |
| 麦克风默认状态  | 标注说明（只读）                | 关闭  | 进入会议时麦克风默认静音          |
| 摄像头默认状态  | 标注说明（只读）                | 关闭  | 进入会议时摄像头默认关闭          |
| 屏幕共享默认状态 | 标注说明（只读）                | 关闭  | 进入会议时屏幕共享默认关闭         |


点击**「确认创建」**按钮后，客户端向服务端发送创建请求，服务端返回唯一的**房间号（Room ID）**，客户端随即进入会议主界面，创建者身份为**主持人（Host）**。

#### 3.2.2 加入会议

用户在主界面点击**「加入会议」**按钮，弹出加入会议面板。

**操作流程：**

1. 用户在输入框中输入**房间号**；
2. 点击**「加入」**按钮，客户端向服务端查询房间信息；
3. 若房间**不存在**，弹出错误提示框："房间不存在，请确认房间号是否正确。"；
4. 若房间**已满员**，弹出提示框："会议已达到最大人数限制，无法加入。"；
5. 若房间**需要密码**，弹出密码输入对话框，用户输入密码后提交验证；
  - 密码错误：弹出错误提示"密码错误，请重新输入。"，可重试；
  - 密码正确：进入会议主界面，身份为**普通参会者（Participant）**；
6. 若房间**无需密码**，直接进入会议主界面。

### 3.3 会议主界面布局

会议主界面采用**左右分栏布局**，结构清晰，操作高效。

```
+------------------------------------------------------------------+
|                        顶部标题栏                                 |
|  [会议标题]   房间号：XXXXXX       [菜单栏：设备切换 / 设置]     |
+------------------------------+---------------------------------+--+
|                              | +-----------------------------+ |
|                              | |       参会人员列表           | |
|                              | |  (可滚动，主持人置顶)        | |
|      左侧视频区域             | |                             | |
|  (纵向排列所有开启摄像头的    | +-----------------------------+ |
|   参会者视频流，支持滚动条)   | +-----------------------------+ |
|                              | |       文字聊天区域           | |
|                              | |  (消息列表 + 输入框)         | |
|                              | |                             | |
|                              | +-----------------------------+ |
+------------------------------+---------------------------------+--+
|                        底部工具栏                                 |
|  [摄像头] [麦克风] [屏幕共享]  采集音量:滑块  播放音量:滑块  [退出] |
+------------------------------------------------------------------+
```

#### 3.3.1 左侧视频区域

- **功能**：实时展示所有已开启摄像头的参会者的视频流；
- **排列方式**：视频块纵向依次排列，每个视频块包含参会者昵称水印（显示于视频左下角）；
- **滚动支持**：当参会人数较多、视频块超出可视区域时，自动显示**垂直滚动条**；
- **无视频状态**：摄像头关闭的参会者不在视频区域中显示，或显示为含默认头像的占位块（可配置）；
- **视频比例**：每个视频块固定保持 16:9 宽高比，宽度自适应左侧面板宽度。

#### 3.3.2 右侧参会人员列表（上半部分）

- **排列规则**：
  - 主持人始终显示在列表**第一位**，以特殊标识（如「主持人」角标或金色边框）加以区分；
  - 其余参会者按**昵称字母序（A-Z）**升序排列；
- **列表项内容**：头像（圆形裁剪）、昵称、当前音视频状态图标（麦克风/摄像头是否开启）；
- **滚动支持**：参会人数超过可视区域时，显示垂直滚动条。

#### 3.3.3 右侧文字聊天区域（下半部分）

- **消息显示区**：纵向滚动的消息列表，新消息追加至底部；
- **消息布局**（每条消息）：

```
  +-----------------------------------------------+
  |  [头像]  昵称                                  |
  |          消息内容文本（支持自动换行）             |
  +-----------------------------------------------+
```

- 头像位于左侧，圆形裁剪，尺寸 40x40 像素；
- 昵称显示于头像右上方，字体较小，颜色为强调色；
- 消息内容位于头像右侧、昵称下方，支持自动换行；
- **消息输入区**：单行或多行文本输入框 + 「发送」按钮，支持按 `Enter` 键发送，`Shift+Enter` 换行；
- **消息长度限制**：单条消息不超过 **1024 字节**；
- **实时广播**：消息发送后由服务端即时转发给房间内所有参会者。

### 3.4 音视频控制

#### 3.4.1 工具栏控件

底部工具栏提供以下控件，从左至右排列：


| 控件     | 类型                  | 默认状态 | 说明                              |
| ------ | ------------------- | ---- | ------------------------------- |
| 摄像头开关  | 图标按钮（Toggle Button） | 关闭   | 点击切换本地摄像头的开启/关闭状态               |
| 麦克风开关  | 图标按钮（Toggle Button） | 关闭   | 点击切换本地麦克风的开启/关闭状态               |
| 屏幕共享开关 | 图标按钮（Toggle Button） | 关闭   | 点击开始/停止屏幕共享；开启后将捕获整个屏幕并推流给其他参会者 |
| 采集音量滑块 | 水平滑块（Slider）        | 75%  | 调节本地麦克风采集音量（0%~100%）            |
| 播放音量滑块 | 水平滑块（Slider）        | 75%  | 调节扬声器播放音量（0%~100%）              |
| 退出会议   | 文字按钮（危险色）           | —    | 退出当前会议；主持人点击则关闭整个会议             |


#### 3.4.2 菜单栏设备切换

顶部菜单栏提供设备管理菜单，包含：

- **切换摄像头**：列出系统中所有可用的视频采集设备，用户可在会议中途切换；
- **切换麦克风**：列出系统中所有可用的音频输入设备，用户可在会议中途切换；
- 切换操作即时生效，无需重新进入会议。

#### 3.4.3 状态同步

- 本地音视频状态变更后，客户端即时通知服务端，服务端将状态广播给房间内其他参会者；
- 参会人员列表中的麦克风/摄像头状态图标实时更新，反映各参会者的当前状态。

### 3.5 文字聊天

#### 功能描述

文字聊天模块仅支持纯文本消息，不支持文件传输、图片发送或表情包。

#### 消息规则


| 规则项  | 说明                               |
| ---- | -------------------------------- |
| 最大长度 | 单条消息不超过 1024 字节（UTF-8 编码）        |
| 发送方式 | 按 `Enter` 键发送；`Shift+Enter` 插入换行 |
| 广播范围 | 房间内所有参会者（包括发送者自己）                |
| 历史消息 | 仅保留本次会议期间的消息，会议结束后不保存            |


#### 消息显示样式

- 每条消息由**头像**、**昵称**、**消息内容**三部分组成；
- 头像位于消息行左侧，圆形裁剪，40x40 像素；
- 昵称以小号加粗字体显示于头像右上方，颜色使用强调色以区分不同发送者；
- 消息内容以正常字号显示于昵称下方，支持自动换行；
- 不同用户的消息之间有适当间距，视觉上清晰区分；
- 消息列表自动滚动至最新消息。

### 3.6 退出会议机制

#### 普通参会者退出

1. 点击工具栏**「退出会议」**按钮；
2. 弹出确认对话框："确定要退出会议吗？"，提供「确定」和「取消」选项；
3. 确认后，本地客户端断开与服务端的连接，界面返回**主界面**；
4. 其余参会者的参会人员列表中，该用户条目自动移除。

#### 主持人退出（关闭会议）

1. 主持人点击工具栏**「退出会议」**按钮；
2. 弹出特殊确认对话框："您是会议主持人，退出将关闭整个会议并移除所有参会者，确认继续吗？"，提供「关闭会议」和「取消」选项；
3. 主持人确认后：
  - 服务端向房间内所有参会者推送"会议已结束"通知；
  - 所有参会者客户端弹出提示框："会议已结束"，点击「确定」后返回主界面；
  - 主持人客户端同步返回主界面；
4. 服务端销毁对应的房间资源。

## 4. 界面设计说明

### 4.1 整体视觉风格

客户端采用**深色现代风格（Dark Modern）**，以深灰色为主色调背景，搭配明亮的蓝色系强调色，营造专业、沉稳的视觉氛围，同时降低长时间使用的视觉疲劳。


| 设计元素  | 规格说明                                  |
| ----- | ------------------------------------- |
| 主背景色  | `#1E1E2E`（深蓝灰）                        |
| 次级背景色 | `#2A2A3E`（面板/卡片背景）                    |
| 强调色   | `#4F8EF7`（蓝色，用于按钮、高亮、链接）              |
| 危险色   | `#E05C5C`（红色，用于退出按钮、错误提示）             |
| 成功色   | `#4CAF82`（绿色，用于连接成功、开启状态）             |
| 主文字色  | `#E8E8F0`（近白色）                        |
| 次级文字色 | `#8888A8`（灰紫色，用于昵称、提示文字）              |
| 圆角半径  | 按钮 8px，卡片 12px，头像圆形（50%）              |
| 阴影    | 面板使用柔和投影：`0 4px 16px rgba(0,0,0,0.4)` |


### 4.2 主界面

- 主界面背景使用主背景色，中央垂直居中显示应用 Logo（简洁线条图标）、产品名称及两个主操作按钮；
- **「创建会议」**按钮使用强调色填充，圆角矩形，宽度 200px；
- **「加入会议」**按钮使用描边样式（透明填充 + 强调色边框），与创建按钮并排或垂直排列；
- 右上角提供**用户设置入口**（头像缩略图 + 昵称），点击弹出设置面板。

### 4.3 会议主界面

- **顶部标题栏**：高度 48px，显示会议标题（左侧）、房间号（中间，可一键复制）、菜单栏（右侧）；背景色略深于主背景；
- **左侧视频区域**：深色背景，视频块之间保留 8px 间距；每个视频块具有 12px 圆角和细微边框；昵称水印使用半透明背景条覆盖于视频左下角；
- **右侧分栏**：使用次级背景色，与左侧视频区域之间有 1px 分隔线；上下两个区域之间有拖动式分隔条（可调整各区域高度）；
- **底部工具栏**：高度 64px，深色背景，控件居中排列；图标按钮使用 SVG 图标，开启/关闭状态以不同颜色填充区分（开启为强调色，关闭为灰色）；滑块使用强调色轨道。

### 4.4 对话框与弹窗

- 所有弹窗采用次级背景色，圆角 16px，带有柔和阴影；
- 弹窗标题使用 18px 加粗字体；内容区使用 14px 正常字重；
- 操作按钮区域位于弹窗底部，主操作按钮（强调色）与次操作按钮（描边样式）左右排列；
- 危险操作弹窗（如关闭会议确认）的主操作按钮使用危险色（红色）。

### 4.5 图标与字体

- 图标：使用 Material Design Icons 或 Fluent UI Icons 风格的 SVG 图标，线条简洁；
- 字体：界面使用系统默认无衬线字体（Windows 上为 Segoe UI），确保跨 DPI 清晰渲染；
- 字号层级：标题 18px、正文 14px、辅助信息 12px。

## 5. 交互流程说明

### 5.1 启动与进入主界面

```
启动应用
  └─→ 加载本地用户配置（昵称、头像）
        └─→ 显示主界面
              ├─→ [设置] → 打开用户信息设置面板 → 修改保存 → 返回主界面
              ├─→ [创建会议] → 打开创建面板 → 配置 → 确认 → 进入会议主界面
              └─→ [加入会议] → 输入房间号 → 验证 → 进入会议主界面
```

### 5.2 创建会议完整流程

```
点击「创建会议」
  └─→ 弹出创建面板
        ├─→ 设置最大人数（SpinBox）
        ├─→ 切换密码开关 → [开启] 显示密码输入框
        └─→ 点击「确认创建」
              └─→ 发送创建请求至服务端
                    ├─→ [成功] 返回 Room ID → 进入会议主界面（主持人）
                    └─→ [失败] 弹出错误提示
```

### 5.3 加入会议完整流程

```
点击「加入会议」 → 输入 Room ID → 点击「加入」
  └─→ 客户端查询服务端
        ├─→ 房间不存在 → 弹出错误提示
        ├─→ 房间已满 → 弹出提示
        ├─→ 需要密码 → 弹出密码输入框
        │     ├─→ 密码错误 → 提示重试
        │     └─→ 密码正确 → 进入会议主界面（参会者）
        └─→ 无需密码 → 进入会议主界面（参会者）
```

### 5.4 音视频设备操作流程

```
点击摄像头按钮
  ├─→ 开启：启动摄像头采集 → 推流 → 视频块出现在左侧区域
```

## 7. 网络通信架构

### 7.1 总体通信模型

云会议系统**仅在局域网环境下运行**，无需公网穿透或 NAT 处理。通信采用**客户端—服务端（C/S）集中转发模型**：所有客户端均与同一台服务端建立连接，媒体流和信令均经由服务端中转，不采用 P2P 直连方式。

```
客户端 A ──┐
客户端 B ──┼──→ 服务端（Ubuntu 24，原生 C++17 POSIX Socket）──→ 转发给其他所有客户端
客户端 C ──┘
```

- **客户端**：使用 Qt 6.9.1 的 Qt Network 模块（`QTcpSocket`、`QUdpSocket`）与服务端通信；
- **服务端**：使用 Linux 原生 C++17 标准库（`<sys/socket.h>`、`<netinet/in.h>` 等 POSIX Socket API）实现，不依赖除json.hpp外的第三方网络库。

---

### 7.2 传输协议选型


| 数据类型               | 传输协议 | 理由                   |
| ------------------ | ---- | -------------------- |
| 信令（创建/加入/退出/状态变更等） | TCP  | 需要可靠、有序投递；数据量小，延迟不敏感 |
| 文字聊天消息             | TCP  | 需要可靠投递，不允许丢失         |
| 音频流                | UDP  | 低延迟优先；局域网丢包率极低，无需重传  |
| 视频流                | UDP  | 低延迟优先；局域网丢包率极低，无需重传  |
| 屏幕共享流              | UDP  | 同视频流                 |


> **设计决策**：由于系统仅在局域网内运行，链路质量稳定，丢包率接近零，因此媒体流**不实现复杂的丢包重传机制（如 NACK/FEC）**，以降低实现复杂度和端到端延迟。

---

### 7.3 端口规划


| 用途        | 协议  | 服务端监听端口 | 说明                  |
| --------- | --- | ------- | ------------------- |
| 信令通道      | TCP | 9000    | 客户端连接后保持长连接         |
| 音频流接收（上行） | UDP | 9001    | 服务端接收各客户端上行音频       |
| 视频流接收（上行） | UDP | 9002    | 服务端接收各客户端上行视频/屏幕共享  |
| 音频流下发（下行） | UDP | 9003    | 服务端向各客户端下发音频        |
| 视频流下发（下行） | UDP | 9004    | 服务端向各客户端下发各路视频/屏幕共享 |


> 端口号均为规划值，可在配置文件中修改。客户端通过 TCP 信令通道获知服务端的媒体端口信息。

---

### 7.4 信令协议设计

#### 7.4.1 通用数据包格式

所有 TCP 信令数据包采用统一的**定长包头 + 变长载荷**格式：

```
+--------+----------+--------------------+
|  类型  | 载荷长度 |    载荷（JSON）    |
| 1 byte | 4 bytes  |      N bytes       |
+--------+----------+--------------------+
```

- **类型（Type）**：标识消息类型（见下表）；
- **载荷长度**：网络字节序（大端）的 `uint32_t`，表示后续 JSON 载荷的字节数；
- **载荷（Payload）**：UTF-8 编码的 JSON 字符串。

客户端侧使用 `QTcpSocket` 读取时，先读取 6 字节包头，解析载荷长度后再读取对应字节数的载荷，避免粘包问题。

### 7.5 媒体流传输设计

#### 7.5.1 音频流

- **编码格式**：使用 FFmpeg `avcodec` 将 PCM 原始音频编码为 **Opus**（低延迟，适合实时通话）；
- **采样率**：48000 Hz，单声道（Mono）；
- **帧长**：每帧 20 ms（960 个采样点）；
- **UDP 包结构**：

```
+----------+----------+----------+--------------------+
| 用户 ID  |  时间戳  |  序列号  | Opus 音频帧数据    |
| 4 bytes  | 4 bytes  | 2 bytes  |     N bytes        |
+----------+----------+----------+--------------------+
```

- 服务端收到各客户端上行音频后，**按房间直接转发**给同房间内其他所有客户端（不做服务端混音）；
- 客户端收到多路音频后，各自解码并由系统音频驱动混合输出；
- 不实现丢包重传；序列号仅用于检测乱序，乱序包直接丢弃。

#### 7.5.2 视频流 / 屏幕共享流

- **编码格式**：使用 FFmpeg `avcodec` 编码为 **H.264**（Baseline Profile）；
- **分辨率**：摄像头默认 1280×720（16:9）；屏幕共享按屏幕实际分辨率等比缩放后发送；
- **帧率**：目标 24 fps；
- **分片策略**：单帧数据若超过 MTU（约 1400 字节），拆分为多个 UDP 包发送；
- **UDP 包结构**：

```
+----------+----------+----------+-------+----------+----------------------+
| 用户 ID  |  时间戳  |  序列号  | 标志  | 分片索引 | H.264 NALU 分片数据  |
| 4 bytes  | 4 bytes  | 2 bytes  |1 byte | 1 byte   |      N bytes         |
+----------+----------+----------+-------+----------+----------------------+
```

- **标志位**：`bit0` = IDR 关键帧标记，`bit1` = 分片起始，`bit2` = 分片结束；
- 服务端按用户 ID 区分来源，**逐路转发**给同房间内其他客户端；
- 不实现 FEC/NACK；关键帧丢失时，客户端通过 TCP 信令发送 `REQUEST_KEYFRAME` 请求，服务端通知对应发送方立即编码一帧 IDR 帧。

### 7.6 连接生命周期管理

#### 7.6.1 连接建立

```
客户端启动
  └─→ QTcpSocket 连接服务端 TCP 9000 端口
        ├─→ 连接成功
        │     └─→ 绑定本地 UDP 端口（随机）
        │           └─→ 通过 TCP 信令将本地 UDP 地址+端口告知服务端
        └─→ 连接失败（5 秒超时）→ 弹出"无法连接到服务器"提示，状态回滚为 Idle，可重试
```

#### 7.6.2 首次连接超时

- 客户端发起 `connectToServer()` 后启动 **5 秒**单次定时器 `m_connectTimer`；
- 若在 5 秒内 TCP 连接建立成功，定时器自动停止；
- 若 5 秒内仍未建立连接，`m_connectTimer` 超时触发 `socket.abort()`，中止连接并 emit `connectTimeout`；
- `MeetingController` 收到 `connectTimeout` 后将状态回滚为 `Idle`，并弹出错误提示"无法连接到服务器，请检查服务器是否已启动或网络是否正常"。

#### 7.6.3 心跳保活

- 客户端每 **5 秒**通过 TCP 信令通道向服务端发送 `HEARTBEAT` 消息；
- 服务端回复 `HEARTBEAT_ACK`；
- 若服务端连续 **3 次**（15 秒）未收到某客户端的心跳，则判定该客户端已断线，主动将其从房间中移除，并广播 `MEMBER_LEAVE` 给其他成员；
- 客户端若在 10 秒内未收到 `HEARTBEAT_ACK`，则尝试重连服务端。

#### 7.6.4 断线重连

- 客户端检测到 TCP 连接断开后，每 **2 秒**尝试重连一次，最多重试 **5 次**；
- 重连成功后，自动重新发送 `JOIN_ROOM` 请求恢复会议状态；
- 5 次重连均失败后，弹出提示"网络连接已断开，请检查局域网连接"，返回主界面。

#### 7.6.5 正常断开

- 用户点击「退出会议」并确认后，客户端先通过 TCP 发送 `LEAVE_ROOM` 信令，再关闭 `QTcpSocket` 和 UDP Socket；
- 主持人退出时发送 `LEAVE_ROOM`（服务端识别其为主持人后广播 `ROOM_CLOSED`），然后关闭连接。

### 7.7 客户端网络模块实现要点（Qt Network）


| 类              | 用途                        |
| -------------- | ------------------------- |
| `QTcpSocket`   | 信令通道的连接、读写、断线检测           |
| `QUdpSocket`   | 音频/视频/屏幕共享流的发送与接收         |
| `QHostAddress` | 解析服务端 IP 地址               |
| `QDataStream`  | 辅助序列化包头字段（大端字节序）          |
| `QTimer`       | 心跳定时器（5 秒间隔）、重连定时器（2 秒间隔） |


**粘包处理**：`QTcpSocket` 的 `readyRead` 信号触发时，维护一个内部接收缓冲区（`QByteArray`），每次尝试从缓冲区头部解析完整数据包（先读 6 字节包头，获取载荷长度，再等待足够字节后解析），未凑齐时留在缓冲区等待下次数据到来。

**线程模型**：网络收发在主线程的 Qt 事件循环中进行（信号/槽驱动），音视频编解码在独立 `QThread` 中运行，通过线程安全队列与网络模块交换数据，避免阻塞 UI。

---

### 7.8 服务端网络实现要点（原生 C++17 + json.hpp）

- 使用 `epoll`（Linux）实现高效的 I/O 多路复用，单线程事件循环处理所有客户端的 TCP 信令读写；
- TCP 监听 socket 设置 `SO_REUSEADDR`，子连接设置 `TCP_NODELAY` 以降低延迟；
- **JSON 解析**：使用 `include/json.hpp`（nlohmann/json 单头文件库）解析客户端发送的 JSON 载荷，无需额外依赖；
  ```cpp
  #include "json.hpp"
  using json = nlohmann::json;
  
  // 解析示例
  json payload = json::parse(json_string);
  int max_members = payload["max_members"];
  std::string nickname = payload["nickname"];
  ```
- UDP 媒体流接收使用独立线程（每个媒体端口一个线程），收到数据包后解析包头中的用户 ID，查询房间成员表，依次向同房间其他成员的 UDP 端点转发；
- 房间数据（成员列表、状态信息）使用 `std::unordered_map` 存储，访问时加 `std::mutex` 保护；
- 服务端仅使用 C++17 标准库、POSIX API 和 json.hpp，不引入其他第三方库。
