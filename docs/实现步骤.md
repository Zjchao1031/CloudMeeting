## 阶段二：客户端业务逻辑层

> 目标：实现本地状态管理与业务规则，UI 与业务层通过信号槽解耦，暂不接入真实网络。

### 2.1 数据模型完善

- 完善 `RoomInfo`：房间号、最大人数、是否有密码、主持人 userId
- 完善 `Participant`：userId、昵称、头像 base64、is_host、摄像头/麦克风/屏幕共享开关状态
- 完善 `ChatMessage`：userId、昵称、头像 base64、消息内容、时间戳

### 2.2 用户资料服务（UserProfileService）

- 昵称与头像本地持久化（`QSettings` 或 JSON 文件写入应用数据目录）
- 应用启动时自动加载，未配置时使用默认昵称和默认头像
- 头像裁剪为正方形并缩放至 64x64 像素
- 未设置头像时加载内置默认头像资源（`:/avatars/default.png`）

### 2.3 参会者仓库（ParticipantRepository）

- `upsertParticipant()`：新增或更新成员信息（以 userId 为 key）
- `removeParticipant()`：移除指定成员
- `sortedParticipants()`：主持人置顶，其余按昵称升序（A-Z）排列
- 成员列表变更时发出 `participantsChanged` 信号，通知 UI 刷新

### 2.4 会议控制器（MeetingController）

- 实现会议状态机（Idle -> Connecting -> InMeeting -> Reconnecting -> Idle）
- `createRoom()`：校验参数，通过 `NetworkFacade` 发出创建请求
- `joinRoom()`：校验房间号非空，通过 `NetworkFacade` 发出加入请求
- `leaveRoom()`：区分主持人（关闭会议）与普通参会者（离开会议）
- `onRoomClosed()`：收到 ROOM_CLOSED 后清理状态并返回主界面
- `onMemberJoin()` / `onMemberLeave()`：更新 `ParticipantRepository`
- `onMediaStateSync()`：更新对应成员的音视频状态

### 2.5 聊天服务（ChatService）

- `validateMessageUtf8Bytes()`：UTF-8 编码后长度大于 1024 字节则返回 false
- `sendMessage()`：校验通过后通过 `NetworkFacade` 发出发送请求
- `appendIncomingMessage()`：追加消息并通知 UI 滚动到底部
- 会议结束时调用 `clearMessages()` 清空缓存

### 2.6 UI 与业务层信号槽绑定

- `MainWindow` 按钮点击 -> 调用 `MeetingController` 方法
- `MeetingController` 状态变更信号 -> 驱动界面切换（主界面 <-> 会议主界面）
- `ParticipantRepository` 变更信号 -> 刷新 `ParticipantListWidget`
- `ChatService` 新消息信号 -> `ChatPanel` 追加消息并滚动
- 用户设置保存 -> `UserProfileService` 写盘 -> 更新主界面右上角显示
- 音视频开关点击 -> `MediaControlModule` 切换状态 -> 发出 MEDIA_STATE 信令（暂用 Mock）

**阶段二交付物**：UI 原型接入本地业务逻辑，本地流程均正常运转，使用 Mock 数据驱动界面。

## 阶段三：客户端网络通信层

> 目标：实现 TCP 信令与 UDP 媒体传输的收发框架，对接真实服务端。

### 3.1 协议编解码（PacketCodec）

- TCP 数据包编码：Type(1 byte) + Length(4 bytes 大端 uint32) + Payload(JSON UTF-8)
- TCP 数据包解码与粘包处理：维护 `QByteArray` 接收缓冲区，循环解析完整包
- UDP 音频包头：userId(4) + timestamp(4) + seq(2) + OpusData(N)
- UDP 视频包头：userId(4) + timestamp(4) + seq(2) + flags(1) + fragIndex(1) + H264Data(N)

### 3.2 TCP 信令客户端（SignalingClient）

- `connectToServer(host, port=9000)`：使用 `QTcpSocket` 建立连接
- `sendPacket(type, jsonPayload)`：序列化包头 + JSON 载荷后写入 socket
- `onReadyRead()`：将新数据追加到缓冲区，调用 `parsePackets()` 循环解析
- 实现各信令消息分发：
  - CREATE_ROOM_ACK (0x02)：通知 MeetingController 创建结果
  - JOIN_ROOM_ACK (0x04)：通知 MeetingController 加入结果
  - ROOM_CLOSED (0x06)：通知 MeetingController 会议已被关闭
  - MEMBER_JOIN (0x07)：通知 ParticipantRepository 新成员加入
  - MEMBER_LEAVE (0x08)：通知 ParticipantRepository 移除成员
  - MEDIA_STATE_SYNC (0x0A)：通知 MeetingController 更新成员音视频状态
  - CHAT_BROADCAST (0x0C)：通知 ChatService 追加新消息
  - HEARTBEAT_ACK (0x0E)：重置心跳超时计时器

### 3.3 心跳与重连（SignalingClient + ReconnectStrategy）

- `startHeartbeat()`：`QTimer` 每 5 秒发送一次 HEARTBEAT 包
- 心跳超时检测：10 秒内未收到 HEARTBEAT_ACK 则触发重连
- `handleReconnect()`：每 2 秒重试一次，最多 5 次；5 次失败后弹出提示并返回主界面
- 重连成功后自动重新发送 JOIN_ROOM 请求恢复会议状态

### 3.4 UDP 媒体传输客户端（MediaUdpClient）

- 绑定本地随机 UDP 端口，通过 TCP 信令将本地 UDP 地址+端口告知服务端
- `sendAudioFrame(data)`：封装音频包头后发送到服务端 UDP 9001 端口
- `sendVideoFrameFragments(data)`：超过 MTU（1400 字节）时分片，发送到 UDP 9002 端口
- `onAudioDatagram()`：监听 UDP 9003 端口，解包后送入音频解码队列
- `onVideoDatagram()`：监听 UDP 9004 端口，按 userId + seq 重组分片，完整帧送入视频解码队列
- 关键帧丢失时通过 TCP 信令发送 REQUEST_KEYFRAME

### 3.5 网络外观层（NetworkFacade）

- 封装 `createRoom()` / `joinRoom()` / `leaveRoom()` / `sendChatMessage()` / `sendMediaState()` 统一接口
- 业务层只与 `NetworkFacade` 交互，不直接引用 `SignalingClient` 或 `MediaUdpClient`

**阶段三交付物**：客户端可与服务端完成 TCP 信令握手、心跳保活、重连恢复；UDP 媒体包收发框架就绪，信令流程端到端可通。

## 阶段四：客户端设备与媒体层

> 目标：接入 FFmpeg 实现音视频采集、编解码与渲染，完成完整的媒体流收发闭环。

### 4.1 设备管理（DeviceManager）

- `enumerateDevices()`：枚举系统中所有可用摄像头与麦克风设备，填充设备列表
- `switchCamera(id)`：已开启时无缝切换到新摄像头；未开启时记录待用设备
- `switchMicrophone(id)`：已开启时无缝切换到新麦克风；未开启时记录待用设备
- 将设备列表同步到顶部菜单栏「设备」菜单（动态构建 `QAction` 列表）

### 4.2 摄像头采集（CameraCaptureStrategy）

- 使用 FFmpeg avformat/avdevice 打开摄像头（Windows 下使用 dshow 输入格式）
- 在独立 `QThread` 中循环读取原始视频帧，推送至编码队列
- 支持随时启停，切换设备时先停止旧采集、再启动新设备

### 4.3 屏幕共享采集（ScreenCaptureStrategy）

- 使用 FFmpeg gdigrab 捕获全屏画面，按 24fps 定时截帧推送至编码队列
- 支持随时启停

### 4.4 视频编码（VideoEncoder，H.264）

- 使用 FFmpeg avcodec 初始化 H.264 编码器（Baseline Profile，1280x720，24fps）
- 编码线程从采集队列取帧，输出 NALU 数据，超过 MTU 时分片发送
- 定期（或收到 REQUEST_KEYFRAME 时）强制编码 IDR 关键帧

### 4.5 音频采集与编码（AudioEncoder，Opus）

- 使用 FFmpeg avdevice 打开麦克风（dshow），采样率 48000 Hz，单声道
- 每 20ms（960 个采样点）一帧，编码为 Opus 后交给 MediaUdpClient 发送
- 音量增益根据采集音量滑块实时调整（0%~100%）

### 4.6 视频解码与渲染（VideoDecoder + VideoTileWidget）

- 从 MediaUdpClient 接收各路视频帧，使用 FFmpeg avcodec H.264 解码
- 使用 swscale 转换为 RGB32，包装为 `QImage` 后通过信号投递到主线程渲染
- 动态管理 `VideoTileWidget` 实例：成员开启摄像头时创建，关闭时移除
- 渲染节流：限制刷新频率不超过 24fps

### 4.7 音频解码与播放（AudioDecoder）

- 从 MediaUdpClient 接收各路音频帧，使用 FFmpeg avcodec Opus 解码为 PCM
- 多路 PCM 数据送入 `QAudioSink` 播放；播放音量根据滑块实时调整
- 乱序包直接丢弃

### 4.8 媒体引擎总控（MediaEngine）

- 统一管理采集、编码、解码、渲染的启停与线程生命周期
- 暴露简洁接口：`startCameraCapture()` / `stopCameraCapture()` / `startScreenShare()` / `stopScreenShare()` / `setCaptureVolume(int)` / `setPlaybackVolume(int)`
- 确保工作线程安全退出，避免资源泄漏

**阶段四交付物**：客户端完整实现音视频采集、H.264/Opus 编解码、视频渲染与音频播放；屏幕共享可采集并推流；设备动态切换正常；媒体延迟满足局域网小于 150ms 目标。

## 阶段五：服务端实现

> 目标：在 Ubuntu 24 上用原生 C++17 + POSIX Socket 实现完整服务端，支持多客户端并发接入。

### 5.1 工程基础

- 配置 `CloudMeetingServer/CMakeLists.txt`，确认 C++17 + POSIX 环境可编译
- 引入 `json.hpp`（nlohmann/json），用于解析客户端 JSON 载荷
- 实现 `Logger`：带时间戳的控制台日志输出
- 实现 `IdGenerator`：生成 6 位随机数字房间号（确保不重复）和唯一用户 ID

### 5.2 TCP 信令服务器（EpollReactor + TcpSignalingServer）

- 实现 `EpollReactor`：基于 epoll 的单线程 I/O 事件循环，监听 TCP 9000 端口，SO_REUSEADDR，TCP_NODELAY
- 实现 `TcpSignalingServer`：维护各客户端接收缓冲区，粘包解析后分发到对应 Handler

### 5.3 信令处理器（Handler 层）

- `CreateRoomHandler`：创建房间，分配房间号，将创建者设为主持人，回复 CREATE_ROOM_ACK
- `JoinRoomHandler`：查询房间是否存在/已满/密码校验；成功则加入，广播 MEMBER_JOIN，回复 JOIN_ROOM_ACK
- `LeaveRoomHandler`：普通成员离开广播 MEMBER_LEAVE；主持人离开广播 ROOM_CLOSED 并销毁房间
- `MediaStateHandler`：广播 MEDIA_STATE_SYNC 给房间内其他成员
- `ChatMessageHandler`：校验消息长度不超过 1024 字节，广播 CHAT_BROADCAST 给房间内所有成员
- `HeartbeatHandler`：回复 HEARTBEAT_ACK；超时 15 秒则主动断线并广播 MEMBER_LEAVE
- `RequestKeyframeHandler`：通知对应发送方客户端立即发送 IDR 关键帧

### 5.4 房间与会话管理

- `RoomManager`：`std::unordered_map` 管理所有活跃房间，增删改查加 `std::mutex` 保护
- `SessionManager`：管理所有 TCP 客户端会话，维护 fd 到会话的映射
- `ClientSession`：存储客户端 fd、userId、所在房间号、UDP 端点地址、最后心跳时间、接收缓冲区
- `Room`：存储房间号、最大人数、密码哈希、成员列表、主持人指针

### 5.5 UDP 媒体转发服务器（UdpMediaServer）

- 独立线程监听 UDP 9001（音频上行）和 UDP 9002（视频上行）端口
- 收到数据包后解析包头中的 userId，查询 RoomManager 找到所在房间
- 将数据包原样转发给同房间内其他所有成员的 UDP 下行端点（9003 音频 / 9004 视频）
- 音频与视频转发分别在独立线程中运行

### 5.6 广播服务（BroadcastService）

- `broadcastToRoom(roomId, excludeUserId, packet)`：向房间内除指定成员外的所有成员发送 TCP 信令包
- `sendToUser(userId, packet)`：向指定成员发送 TCP 信令包

**阶段五交付物**：服务端可运行，支持多房间并发、心跳检测、成员加入/离开广播、聊天转发、媒体 UDP 透传；通过命令行工具可手动验证信令流程。

## 阶段六：客户端与服务端联调

> 目标：将客户端与服务端拉通，端到端验证所有功能流程。

### 6.1 信令流程联调

- 联调创建会议：CREATE_ROOM -> CREATE_ROOM_ACK -> 客户端进入会议主界面
- 联调加入会议（无密码/有密码/房间不存在/已满）全路径
- 联调普通参会者退出：LEAVE_ROOM -> 广播 MEMBER_LEAVE -> 其他客户端更新列表
- 联调主持人关闭会议：广播 ROOM_CLOSED -> 所有客户端弹窗并返回主界面
- 联调聊天消息广播实时性
- 联调心跳保活与断线重连恢复
- 联调音视频状态同步（参会人员列表图标实时更新）

### 6.2 音视频媒体流联调

- 联调音频流全链路（采集->编码->UDP上行->转发->UDP下行->解码->播放）
- 联调视频流（H.264 编解码 + 视频渲染）
- 联调屏幕共享采集与推流
- 验证 3~4 路并发音视频稳定性
- 验证 REQUEST_KEYFRAME 关键帧恢复机制

### 6.3 异常场景验证

- 网络中断 3~5 秒后自动重连，会议状态恢复
- 客户端崩溃后服务端 15 秒内检测超时并广播 MEMBER_LEAVE
- 5 次重连失败后客户端弹出提示并返回主界面
- 参会人数达上限时新加入者收到已满提示

**阶段六交付物**：所有流程端到端联调通过，异常场景有合理处理，多客户端并发稳定。

## 阶段七：完善与收尾

### 7.1 性能与稳定性优化

- 验证视频编码帧率不低于 24fps（720p）

## 关键风险与注意事项

| 风险点 | 说明 | 应对策略 |
|--------|------|----------|
| FFmpeg 多路音频输出 | 多路音频解码后混音输出可能有卡顿 | 使用 QAudioSink 管理输出缓冲区 |
| 视频分片重组 | UDP 乱序导致分片无法重组 | 按 seq 排序，超时丢弃并发送关键帧请求 |
| epoll 并发安全 | 多线程访问房间数据结构 | 所有房间/会话数据访问均加 std::mutex 保护 |
| Windows dshow 采集延迟 | 部分摄像头驱动延迟较高 | 调整 FFmpeg avformat 采集参数 |
| 高 DPI 视频渲染 | QImage 在高 DPI 下可能模糊 | 使用 devicePixelRatio() 适配或改用 QOpenGLWidget |
## 阶段六：客户端与服务端联调

> 目标：将客户端与服务端拉通，端到端验证所有功能流程。

### 6.1 信令流程联调

- 联调创建会议：CREATE_ROOM -> CREATE_ROOM_ACK -> 客户端进入会议主界面
- 联调加入会议（无密码/有密码/房间不存在/已满）全路径
- 联调普通参会者退出：LEAVE_ROOM -> 广播 MEMBER_LEAVE -> 其他客户端更新列表
- 联调主持人关闭会议：广播 ROOM_CLOSED -> 所有客户端弹窗并返回主界面
- 联调聊天消息广播实时性
- 联调心跳保活与断线重连恢复
- 联调音视频状态同步（参会人员列表图标实时更新）

### 6.2 音视频媒体流联调

- 联调音频流全链路（采集->编码->UDP上行->转发->UDP下行->解码->播放）
- 联调视频流（H.264 编解码 + 视频渲染）
- 联调屏幕共享采集与推流
- 验证 3~4 路并发音视频稳定性
- 验证 REQUEST_KEYFRAME 关键帧恢复机制

### 6.3 异常场景验证

- 网络中断 3~5 秒后自动重连，会议状态恢复
- 客户端崩溃后服务端 15 秒内检测超时并广播 MEMBER_LEAVE
- 5 次重连失败后客户端弹出提示并返回主界面
- 参会人数达上限时新加入者收到已满提示

**阶段六交付物**：所有流程端到端联调通过，异常场景有合理处理，多客户端并发稳定。
