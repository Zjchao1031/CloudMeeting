> 文档版本：v1.1  

> 基准文档：《[云会议说明文档.md](http://云会议说明文档.md)》v1.0  

> 目标平台：Windows 11 x64  

> 技术基线：C++17 / Qt 6.9.1 / CMake 3.16+ / FFmpeg

---

## 1. 文档概述

### 1.1 目的

本文档在功能说明文档基础上，定义云会议桌面客户端的技术架构、模块边界、核心类设计与并发模型，作为客户端开发、联调、测试与后续扩展的统一技术依据。

### 1.2 范围

本文档覆盖客户端系统设计，包括：

- UI 层、业务层、网络层、设备与媒体层；

- TCP 信令与 UDP 媒体传输设计；

- 音视频控制、聊天、会议管理、断线重连；

- 核心设计模式落地方案。

不包含服务端内部实现细节（仅定义客户端接口与交互约束）。

### 1.3 读者对象

- C++/Qt 客户端开发工程师

- 音视频开发工程师

- 测试工程师

- 架构师/项目负责人

## 2. 总体架构设计

### 2.1 分层架构

客户端采用四层架构，遵循“UI 不直连底层 I/O、业务不耦合具体编解码实现”的原则：

1. **UI 层（Presentation）**

   - 主界面、创建/加入会议弹窗、会议主界面（视频区/成员区/聊天区/工具栏）

   - 只处理用户交互和展示，不直接操作 socket/codec

2. **业务逻辑层（Application/Domain）**

   - 会议生命周期（创建、加入、退出、房间关闭）

   - 成员状态管理（主持人、音视频状态、排序规则）

   - 聊天消息校验与分发（1024 字节限制）

3. **网络通信层（Transport）**

   - TCP 信令通道：创建/加入/离开、状态同步、聊天、心跳、重连

   - UDP 媒体通道：音频/视频/屏幕共享包收发

   - 定长包头 + 变长 JSON 载荷解析（粘包处理）

4. **设备与媒体层（Device & Media）**

   - 摄像头/麦克风枚举与切换

   - 音视频采集、H.264/Opus 编解码、渲染数据分发

   - 屏幕共享采集策略

---

### 2.2 模块划分与职责

- **UserProfileModule**：昵称/头像本地持久化（启动加载、设置保存）

- **MeetingModule**：创建/加入/退出会议、主持人关闭会议逻辑

- **MemberModule**：参会者列表维护、主持人置顶、其余按昵称字母序

- **ChatModule**：消息发送校验（<=1024 bytes UTF-8）、广播展示

- **MediaControlModule**：摄像头/麦克风/屏幕共享开关、采集/播放音量调节

- **DeviceModule**：设备枚举与动态切换

- **SignalingModule(TCP)**：信令收发、心跳、重连、包解析

- **MediaTransportModule(UDP)**：媒体包发送/接收、分片重组、关键帧请求

- **CodecModule(FFmpeg)**：H.264/Opus 编解码封装

- **RenderModule**：视频帧渲染与 UI 刷新节流

---

### 2.3 关键技术栈与约束

- **语言与框架**：C++17 + Qt 6.9.1

- **构建**：CMake 3.16+

- **通信**：`QTcpSocket` + `QUdpSocket`

- **媒体**：FFmpeg（`avcodec/avformat/avutil/swscale/swresample`）

- **协议约束（与说明文档一致）**：

  - TCP：信令 + 聊天

  - UDP：音频/视频/屏幕共享

  - 首次连接超时：5 秒内未建立 TCP 连接则中止并提示用户

  - 心跳：5 秒一次；客户端 10 秒未收 ACK 触发重连

  - 重连：每 2 秒一次，最多 5 次

---

### 2.4 并发模型

采用“事件驱动 + 工作线程”的混合模型：

- **主线程（Qt 事件循环）**

  - UI 绘制与交互

  - `QTcpSocket/QUdpSocket` 的 readyRead 等事件分发

  - 轻量状态更新

- **媒体工作线程**

  - Qt Multimedia 后台线程：音视频采集（`QMediaCaptureSession`）

  - 采集回调中执行编码（H.264/Opus），独立于主线程

  - UDP 媒体数据的发送与接收

- **跨线程通信**

  - Qt 信号槽（`Qt::QueuedConnection`）：跨线程 UI 更新

  - `QMetaObject::invokeMethod`：在主线程安全更新 UI

- **服务端业务处理层** 对应 `CloudMeetingServer/include/handler`、`CloudMeetingServer/include/service`
- **服务端房间与会话数据层** 对应 `CloudMeetingServer/include/domain` 与 `CloudMeetingServer/src/domain`

通过上述目录划分，可以将系统设计文档中的分层架构直接映射到工程代码结构，便于开发任务拆分、代码评审与后续维护。

## 4.2 会议与成员业务层

### 4.2.1 模块职责

- 会议状态机（空闲/连接中/会议中）

- 创建/加入/退出会议流程编排

### 4.2.2 核心类设计

#### `MeetingController`

- 属性：

  - `MeetingState state`

  - `RoomInfo currentRoom`

- 方法：

  - `createRoom(CreateRoomOptions)`

  - `joinRoom(JoinRoomOptions)`

  - `leaveRoom()`

  - `onRoomClosed()`

#### `ParticipantRepository`

- 属性`QHash<QString, Participant> participants`

- 方法：

  - `sortedParticipants()`

### 4.2.3 关键逻辑

- 主持人退出触发“关闭会议”分支

- 普通用户退出仅移除自身并返回主界面

- 成员状态变化（麦克风/摄像头）实时反映到 UI

### 4.2.4 设计模式

- **状态模式**：封装 `Idle/Connecting/InMeeting/Reconnecting` 行为差异，避免 if-else 膨胀

- **外观模式**：`MeetingController` 对 UI 提供统一入口，屏蔽网络与媒体细节

## 4.4 网络通信层（TCP/UDP）

### 4.4.1 模块职责

- TCP 信令连接、封包解包、心跳、重连

- UDP 媒体包收发、分片组装与关键帧请求

### 4.4.2 核心类设计

#### `SignalingClient`

- 属性：

  - `QTcpSocket socket`

  - `QByteArray recvBuffer`

  - `QTimer heartbeatTimer`

  - `QTimer connectTimer`（首次连接超时 5s）

  - `QTimer reconnectTimer`

  - `int reconnectAttempts`

- 方法：

  - `connectToServer()`

  - `sendPacket(uint8_t type, QJsonObject payload)`

  - `onReadyRead()`

  - `parsePackets()`

  - `startHeartbeat()`

  - `handleReconnect()`

  - `onConnectTimeout()`

#### `MediaUdpClient`

- 属性`QUdpSocket audioUdp`, `videoUdp`

- 方法：

  - `sendAudioFrame()`

  - `sendVideoFrameFragments()`

  - `onAudioDatagram()`

  - `onVideoDatagram()`

### 4.4.3 关键逻辑

- TCP 包格式严格遵循说明文档：`Type(1) + Length(4) + Payload(N)`

- 粘包处理：缓冲区循环解析完整包，不完整包留待下次

- 首次连接超时：`connectToServer()` 后启动 5s 定时器，超时则 `socket.abort()` 并 emit `connectTimeout`，`MeetingController` 回滚状态并提示用户

- 关键帧丢失时发送 `REQUEST_KEYFRAME` 信令

### 4.4.4 设计模式

- **策略模式**：`ReconnectStrategy` 定义重连参数（间隔 2s、最多 5 次），逻辑内联于头文件，由 `SignalingClient` 直接引用

- **外观模式**：`NetworkFacade` 对业务层暴露统一接口（create/join/chat/state），并在收到 ACK 后按服务端下发端口初始化 UDP 媒体客户端

## 4.5 设备与媒体层

### 4.5.1 模块职责

- 管理摄像头/麦克风设备切换

- 采集、编码、解码与渲染数据供给

### 4.5.2 核心类设计

#### `DeviceManager`

- 属性：

  - `QList<DeviceInfo> cameras`

  - `QList<DeviceInfo> microphones`

  - `QString currentCameraId`

  - `QString currentMicId`

- 方法：

  - `enumerateDevices()`

  - `switchCamera(QString id)`

  - `switchMicrophone(QString id)`

#### `MediaEngine`

- 属性：

  - `std::unique_ptr<IVideoCaptureStrategy> cameraCaptureStrategy`

  - `std::unique_ptr<IVideoCaptureStrategy> screenCaptureStrategy`

  - `VideoEncoder cameraVideoEncoder`

  - `VideoEncoder screenVideoEncoder`

  - `AudioEncoder audioEncoder`

- 方法：

  - `startCameraCapture()`

  - `stopCameraCapture()`

  - `startScreenShare()`

  - `stopScreenShare()`

  - `setCaptureVolume(int)`

  - `setPlaybackVolume(int)`

### 4.5.3 关键逻辑

- 摄像头与屏幕共享支持并行开启，客户端可同时采集摄像头画面与桌面画面，并分别编码后通过独立视频流发送

- 切换设备时，已开启状态下无缝切换；未开启仅更新“下次生效设备”

- 目标帧率 24fps，过载时采用降帧保护 UI 流畅性

### 4.5.4 设计模式

- **工厂模式**：`CodecFactory` 定义了静态创建编码器/解码器的工厂方法，隔离 FFmpeg 初始化细节；`MediaEngine` 当前通过 `std::make_unique<>` 直接创建实例

- **策略模式**：摄像头采集与屏幕共享采集分别通过 `CameraCaptureStrategy` 和 `ScreenCaptureStrategy` 封装，支持两类采集策略独立启停与并行运行
